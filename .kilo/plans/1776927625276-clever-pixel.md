# Plan de refonte backend MindMesh (Express + CrewAI + Auth robuste)

## Statut d'execution (mise a jour)

Etat observe dans le repository:
- Refactor backend demarre: `backend/public-api/**` cree.
- Orchestrateur prive cree: `backend/orchestrator/**` cree.
- Point d'entree legacy redirige vers la nouvelle API (`server.js`).
- Front adapte a `VITE_API_BASE_URL` + Bearer token.
- `.env.example` et `package.json` enrichis.
- Dependencies npm installees (`package-lock.json` modifie).

Implication:
- Le plan n'est plus theorique: il est **partiellement implemente**.
- Le prochain bloc de travail prioritaire est la **stabilisation** (tests, verification runtime, hardening prod, doc d'exploitation).

## 1) Audit backend actuel

### 1.1 Ce qui existe déjà
- Backend monolithique dans `server.js` (Express, ESM).
- Endpoints consommés par le front:
  - `POST /api/chat` (SSE stream texte via OpenAI SDK).
  - `POST /api/transcribe` (upload audio via `multer`, appel ASR distant).
- Front React déjà fonctionnel (stream SSE, envoi audio, UI complète).
- Variables d'environnement présentes (`OPENAI_API_KEY`, `OPENAI_MODEL`, `PORT`, `VITE_CLERK_PUBLISHABLE_KEY`).
- Nettoyage de base des messages (`role`, `content`) côté chat.

### 1.2 Failles / risques (sécurité, robustesse, scalabilité, dette)
| Catégorie | Constat actuel | Risque | Impact |
|---|---|---|---|
| AuthN/AuthZ | Aucune validation JWT sur `/api/chat` et `/api/transcribe` | Appels anonymes, abus de clé IA | Très élevé |
| CORS | `cors()` ouvert globalement | Origines non maîtrisées | Élevé |
| Upload audio | Pas de limite stricte taille/type dans `multer` | DoS stockage/mémoire, formats inattendus | Élevé |
| I/O fichiers | Usage `fs.*Sync` | Blocage event loop sous charge | Moyen/Élevé |
| Rate limiting | Absent | Saturation API, explosion coûts | Élevé |
| Validation input | Validation minimale | Payload malformés, erreurs runtime | Moyen |
| Gestion erreurs | Contrat non normalisé (JSON/SSE mix) | UX instable, debug difficile | Moyen |
| Observabilité | Logs console simples, pas de corrélation | Incident response lente | Élevé |
| Résilience externe | Dépendance provider sans fallback/circuit breaker | Pannes en cascade | Moyen |
| Architecture | `App.tsx` très couplé au backend actuel + URL locale hardcodée | Déploiement fragile | Élevé |

### 1.3 Gaps fonctionnels
- Pas d’orchestration multi-agent réelle côté backend (CrewAI absent).
- Pas de séparation claire: API publique vs logique d’orchestration.
- Pas de mécanisme service-to-service sécurisé.
- Pas de persistance de conversation/session (optionnel mais utile audit/replay).
- Pas de contrat API versionné/typé côté backend.

## 2) Architecture cible recommandée

### Option A: backend public unique + orchestrateur privé
- **Public API (Node/Express)**: expose uniquement endpoints front (`/api/chat`, `/api/transcribe`), valide JWT utilisateur, applique limites/sécurité, puis délègue au service privé.
- **Orchestrateur privé (FastAPI + CrewAI)**: non exposé internet, reçoit appels internes signés court-terme, exécute workflow agents, appelle modèles externes.

### Option B: double backend public
- Deux backends exposés (ex: Node public + FastAPI public), avec auth et politiques de sécurité dupliquées sur chaque surface.

### Comparaison
| Critère | Option A (API publique + orchestrateur privé) | Option B (double backend public) |
|---|---|---|
| Complexité implémentation | Moyenne (2 services, 1 surface publique) | Élevée (2 surfaces publiques, politiques dupliquées) |
| Coût ops | Maîtrisé | Plus élevé |
| Sécurité | Forte (surface d’attaque réduite) | Plus faible (surface élargie) |
| Time-to-market | Rapide | Plus lent |
| Maintenabilité | Bonne (responsabilités nettes) | Moyenne (drift entre services) |
| Compat front existant | Excellente (contrat inchangé) | Variable (risque adaptation front) |

### Recommandation finale
- **Recommandation: Option A** pour votre contexte (frontend déjà en place, besoin de livraison rapide, sécurité prioritaire).
- Rationale: minimum viable robuste, sans rupture front, avec trajectoire claire vers montée en charge.

## 3) Plan de migration détaillé (incrémental, sans casse front)

### Phase 0 — Stabilisation immédiate (J1-J2)
- Ajouter couche sécurité minimale sur backend actuel:
  - `helmet`, CORS whitelist, rate limit, validation payload.
  - standardisation erreurs JSON/SSE.
- Externaliser URL backend côté front via `VITE_API_BASE_URL`.
- Conserver strictement les endpoints et formats actuels.

### Phase 1 — Strangler pattern backend (J3-J5)
- Créer `backend/public-api` (Express modulaire) sans couper `server.js` legacy.
- Introduire middleware JWT (JWKS) + middleware `request_id` + logs structurés.
- Mettre flag de routage: `ORCHESTRATION_MODE=legacy|hybrid|crewai`.

### Phase 2 — Orchestrateur CrewAI privé (J5-J8)
- Créer `backend/orchestrator` (FastAPI + CrewAI) avec endpoint privé interne.
- Implémenter token interne court (service-to-service).
- Adapter `/api/chat` pour déléguer à l’orchestrateur en mode `hybrid` (fallback legacy).

### Phase 3 — Transcription robuste (J7-J9)
- Durcir `/api/transcribe`: limites upload, MIME whitelist, nettoyage asynchrone, timeout provider.
- Option: mode async + polling si latence ASR élevée (sans imposer immédiatement au front).

### Phase 4 — Durcissement prod & rollout progressif (J10-J12)
- Passage trafic progressif vers mode `crewai` (10% → 50% → 100%).
- Surveillance métriques erreurs/latence/coût.
- Maintien fallback immédiat vers `legacy`.

### Phase 5 — Nettoyage (J13-J14)
- Retirer code obsolète validé, finaliser docs runbook, playbook incident.

### Contrats API à conserver / adapter
- **Conserver**
  - `POST /api/chat` body: `{ messages: [{ role, content }] }`
  - `POST /api/transcribe` multipart `audio`
  - Réponse chat en SSE + event `done`
- **Adapter sans rupture**
  - Header `Authorization: Bearer <jwt>` requis en prod.
  - Erreurs normalisées (`{ error: { code, message, request_id } }`).
  - Ajouter `X-Request-Id` en réponse.

### Stratégie rollback
- Toggle runtime `ORCHESTRATION_MODE` pour revenir en `legacy` en < 1 min.
- Déploiement blue/green pour API publique.
- Health checks bloquants avant promotion de version.
- Journaliser décision de rollback + fenêtre d’analyse post-mortem.

## 4) Conception auth complète

### 4.1 Validation JWT utilisateur (frontend → API publique)
- Source token: `Authorization: Bearer`.
- Vérification signature via **JWKS** (cache local + refresh périodique).
- Vérifications obligatoires claims:
  - `iss` (issuer exact attendu).
  - `aud` (audience API backend).
  - `exp` (non expiré).
  - `nbf` (actif), avec leeway max 60s.
  - `sub` (identifiant utilisateur non vide).
- Vérifications recommandées:
  - `jti` anti-replay (cache court si faisable).
  - `azp`/`client_id` si provider le permet.
- Enrichissement `req.user` minimal: `{ sub, roles, scopes, tenant? }`.

### 4.2 RBAC / permissions minimales
- Rôles minimaux:
  - `user`: accès `chat:write`, `transcribe:write`.
  - `admin` (optionnel): endpoints ops internes non sensibles.
- Principes:
  - deny-by-default.
  - scopes vérifiés par endpoint.

### 4.3 Token interne court (API publique → orchestrateur CrewAI)
- Format: JWT interne signé (clé dédiée, `kid` versionné).
- Claims minimales:
  - `iss=public-api`
  - `aud=crewai-orchestrator`
  - `sub=<user_sub>`
  - `scope=orchestrate:invoke`
  - `exp` TTL 60-120s max
  - `jti` unique
  - `request_id`
- Rotation clé:
  - 2 clés actives (current/next), rotation mensuelle.
  - fenêtre de recouvrement 24h.
  - stockage secret via manager central (pas en dur).

### 4.4 Gestion erreurs 401 / 403 / 429
- `401 Unauthorized`:
  - token absent/invalide/expiré.
  - réponse: code stable (`AUTH_INVALID_TOKEN`), sans fuite d’info.
- `403 Forbidden`:
  - token valide mais scope/role insuffisant (`AUTH_FORBIDDEN`).
- `429 Too Many Requests`:
  - quotas IP + user + endpoint.
  - header `Retry-After` + code `RATE_LIMITED`.
- Logging sécurité:
  - log structuré, anonymisé, corrélé `request_id`, sans dump token.

### 6.2 Middlewares clés
- `request-id`: génère/propage UUID corrélé.
- `auth-jwt`: validation JWKS + claims strictes.
- `authorize-scope`: contrôle permissions par route.
- `rate-limit`: quotas IP/user/endpoint.
- `error-handler`: mapping erreurs -> contrat unique.
- `security`: `helmet`, CORS strict, body limits.

### 6.3 Endpoints clés
- Public API:
  - `POST /api/chat` (SSE, protégé JWT)
  - `POST /api/transcribe` (multipart, protégé JWT)
  - `GET /healthz`, `GET /readyz`
- Orchestrateur privé:
  - `POST /internal/orchestrate` (token S2S requis)
  - `GET /internal/healthz`

### 6.4 Observabilité
- Logs JSON structurés avec champs standard: `timestamp`, `level`, `request_id`, `user_sub`, `route`, `latency_ms`, `upstream_status`.
- Corrélation cross-service via `request_id` transmis dans header interne.
- Métriques minimales:
  - `http_requests_total`
  - `http_request_duration_ms`
  - `auth_failures_total`
  - `orchestrator_calls_total`
  - `provider_errors_total`
- Alertes minimales:
  - taux `5xx` > seuil,
  - `401/403` anormal,
  - latence p95 dégradée.

### 6.5 Tests minimaux (unit + intégration)
- Unit (Node): validation JWT, RBAC, schémas payload, mapping erreurs.
- Unit (Python): vérification token interne, résolution workflow CrewAI.
- Intégration:
  - `POST /api/chat` avec token valide/invalide/expiré.
  - stream SSE jusqu’à event `done`.
  - `POST /api/transcribe` avec fichier valide/invalide/trop volumineux.
  - fallback `legacy` si orchestrateur indisponible (mode hybrid).

## 7) Risques restants + next steps

### Top 5 risques résiduels
1. Dépendance fournisseur IA/ASR externe (latence, indisponibilité, coûts).
2. Prompt injection impactant outils agents si garde-fous insuffisants.
3. Mauvaise gestion rotation clés JWT interne (erreurs auth inter-service).
4. Surcharge en pics sans file d’attente/backpressure.
5. Drift contrats SSE entre API publique et front si tests incomplets.

- J1: sécuriser API actuelle (JWT + CORS + rate limit + request_id).

- J6-J7: brancher `/api/chat` en mode hybrid + tests E2E critiques.

1. Provider exact du JWT front en production (Clerk confirmé ?) + `iss`/`aud` définitifs.

3. SLO attendu (RPS, p95 latency) pour calibrer rate limits et sizing.
4. Exigences conformité (rétention logs/transcriptions, RGPD, anonymisation).
