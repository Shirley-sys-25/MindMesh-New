# Guide d'execution J12-J14 (non local)

## Objectif

Ce plan te guide pour realiser ce qui n'est pas executable uniquement en local:
- **J12**: rollout progressif reel `10% -> 50% -> 100%` avec monitoring.
- **J13**: bascule finale en `ORCHESTRATION_MODE=crewai` avec gate Go/No-Go.
- **J14**: nettoyage legacy apres stabilisation confirmee.

Le code est deja pret; ici on execute la phase **ops/deploiement**.

---

## 0) Preconditions (a valider avant J12)

- Version backend actuelle deployee (celle avec:
  - `ORCHESTRATION_CREWAI_PERCENT`,
  - `/readyz` degrade-aware,
  - fallback `hybrid`,
  - metriques `/metrics`).
- Auth prod active:
  - `AUTH_REQUIRED=true`
  - `AUTH_BYPASS=false`
  - `AUTH_STRICT_SCOPES=true`
- Orchestrateur deploye et accessible depuis l'API publique.
- Dashboard/alerting disponibles (Prometheus/Grafana ou equivalent).
- Rollback teste au moins 1 fois sur staging.

KPIs de gate (issus du runbook):
- `5xx <= 2%` sur 30 min
- p95 `/api/chat` <= `+50%` baseline
- pas de derive continue de:
  - `mindmesh_orchestrator_calls_total{status="failed"}`
  - `mindmesh_provider_errors_total`
- `GET /readyz` stable

---

## 1) Baseline (avant 10%)

Fenetre: 30-60 min minimum.

1. Garder:
   - `ORCHESTRATION_MODE=hybrid`
   - `ORCHESTRATION_CREWAI_PERCENT=0`
2. Capturer baseline:
   - trafic moyen,
   - p95 chat,
   - taux 5xx,
   - 401/403/429,
   - erreurs provider.
3. Verifier endpoints:

```bash
curl -i http://<api-host>/healthz
curl -i http://<api-host>/readyz
curl -s http://<api-host>/metrics
```

4. Figer un point de comparaison (tableau simple `baseline.md` interne ou ticket ops).

---

## 2) J12 - Rollout progressif reel

## Etape A: 10%

Configurer:
- `ORCHESTRATION_MODE=hybrid`
- `ORCHESTRATION_CREWAI_PERCENT=10`

Observer 30-60 min:
- p95 chat
- 5xx
- `mindmesh_orchestrator_calls_total`
- `mindmesh_provider_errors_total`
- `readyz`

**Passage etape suivante si tous les KPIs sont verts.**

## Etape B: 50%

Configurer:
- `ORCHESTRATION_CREWAI_PERCENT=50`

Observer 30-60 min, memes KPIs.

**Passage etape suivante si KPIs verts.**

## Etape C: 100% hybride

Configurer:
- `ORCHESTRATION_CREWAI_PERCENT=100`

Le fallback legacy reste actif car mode `hybrid`.
Observer 60 min (ou plus selon ton trafic).

---

## 3) J13 - Bascule 100% crewai

Condition d'entree (Go/No-Go):
- etape 100% hybrid stable,
- aucun alerting critique,
- rollback valide et documente.

Action:
- `ORCHESTRATION_MODE=crewai`
- garder `ORCHESTRATION_CREWAI_PERCENT=100` (sans effet en mode crewai, mais explicite).

Verification immediate:

```bash
curl -i http://<api-host>/readyz
curl -s http://<api-host>/metrics
```

Verification fonctionnelle (avec token reel):

```bash
curl -N -X POST "http://<api-host>/api/chat" \
  -H "Authorization: Bearer <jwt>" \
  -H "Content-Type: application/json" \
  -d '{"messages":[{"role":"user","content":"Bonjour"}]}'
```

Surveiller 60 min minimum avec seuils runbook.

---

## 4) J14 - Nettoyage legacy final

A faire **uniquement** apres stabilisation crewai (ex: 7 jours sans incident majeur):

1. Inventory legacy restant:
   - chemins fallback `legacy` dans `/api/chat`
   - code provider direct non necessaire
   - variables env legacy non utilisees
2. Retrait progressif:
   - supprimer fallback si non requis par politique ops
   - simplifier logique de routage
3. Revalider:
   - tests API + orchestrateur
   - smoke prod (`healthz`, `readyz`, chat SSE, transcribe)
4. Mettre a jour doc:
   - `RUNBOOK.md`
   - procedure incident/rollback finale

## 5) Commandes de deploiement selon plateforme

## Option A - Kubernetes

```bash
# 10%
kubectl -n <ns> set env deploy/<public-api-deploy> ORCHESTRATION_MODE=hybrid ORCHESTRATION_CREWAI_PERCENT=10
kubectl -n <ns> rollout status deploy/<public-api-deploy>

# 50%
kubectl -n <ns> set env deploy/<public-api-deploy> ORCHESTRATION_CREWAI_PERCENT=50
kubectl -n <ns> rollout status deploy/<public-api-deploy>

# 100% hybrid
kubectl -n <ns> set env deploy/<public-api-deploy> ORCHESTRATION_CREWAI_PERCENT=100
kubectl -n <ns> rollout status deploy/<public-api-deploy>

# crewai strict
kubectl -n <ns> set env deploy/<public-api-deploy> ORCHESTRATION_MODE=crewai ORCHESTRATION_CREWAI_PERCENT=100
kubectl -n <ns> rollout status deploy/<public-api-deploy>
```

## Option B - Docker Compose

Mettre a jour les variables d'env du service API puis:

```bash
docker compose up -d --build <public-api-service>
docker compose ps
```

## Option C - VM/Systemd

Mettre a jour le fichier d'env (ex: `/etc/mindmesh/public-api.env`) puis:

```bash
sudo systemctl daemon-reload
sudo systemctl restart <public-api-service>
sudo systemctl status <public-api-service>
```

---

## 6) Rollback operationnel (<1 min)

Rollback prioritaire:

1. Basculer en legacy:
   - `ORCHESTRATION_MODE=legacy`
2. Redeployer/restart service API.
3. Verifier:

```bash
curl -i http://<api-host>/healthz
curl -N -X POST "http://<api-host>/api/chat" \
  -H "Authorization: Bearer <jwt>" \
  -H "Content-Type: application/json" \
  -d '{"messages":[{"role":"user","content":"test rollback"}]}'
```

Rollback partiel possible:
- rester en `hybrid`
- reduire `ORCHESTRATION_CREWAI_PERCENT` a `10`, puis `0`.
